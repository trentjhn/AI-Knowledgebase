## Problem statement

Our /cook + magnum-opus + consulting-playbook system optimizes for *artifact production* (substantive scaffold, audit triad, agent fleet, security baseline). It under-emphasizes *theory production* — the operator's mental model of how the system fits together. Per Peter Naur (1985), "the program is the theory in the programmer's head; the code is its shadow." AI generates the shadow on demand; theory must be deliberately built.

Symptom: the consulting playbook covers tier-based shipping, audit discipline, decision logs — but doesn't force the operator to articulate **where state lives, where feedback lives, what breaks if a component is deleted** before any code is written. These three questions are the core systems-thinking diagnostic from the video.

Symptom 2: /cook Phase 1.5 walks the 7-property spec framework (WHAT) but doesn't enforce WHY. Specs without WHY decay into requirements lists that don't survive scope debates.

Symptom 3: the consulting playbook authority block covers "what Claude can decide autonomously" but not "what Claude is *capable* of deciding well" — the jagged frontier. Some decisions should be operator-only not because of authority but because of capability boundary.

## Goals

1. **Force theory-building upfront** via the Three Questions framework, baked into both Magnum Opus and consulting playbook (and into design.md as required sections).
2. **Reframe the doc spine as theory-building artifacts**, not bureaucracy. Per Peter Naur framing.
3. **Sharpen authority block with jagged-frontier awareness** — what NOT to delegate even within autonomous scope.
4. **Mandate design-before-prompt discipline** with concrete artifact (sketches in `docs/plans/sketches/`).
5. **Add anti-pattern A17 (system incoherence)** — the actual failure mode from the video's 7000-line file story.
6. **Document AI decision points explicitly** — every place AI output drives runtime behavior gets a verification mechanism declared.
7. **Add System Coherence Audit hunt category** — not file-size limits; checks for files containing multiple unrelated concerns.
8. **Enforce WHY alongside WHAT** in /cook Phase 1.5 spec framework.
9. **Add client involvement model question** to Phase 0.6 engagement discovery.
10. **Reframe audit-agent posture** as "patient senior dev" collaborator, not grader.

All 10 land in **both** Magnum Opus (universal) and consulting playbook (specialized). Magnum Opus gets the universal framing; consulting playbook gets the consulting-specific application.

## Non-goals (deliberately out of scope)

- Hard max-file-length lint gates (rejected — coherence is the signal, not size).
- Mandating sketch tools (Excalidraw, draw.io, etc.) — keep artifact format flexible (markdown OK, ASCII art OK, photo of whiteboard OK).
- Backporting to existing engagements (brett-gove-intell stays as-is).
- Operator-personal-discipline practices (rewrite-AI-code-by-hand-weekly) — out of scope for system rules.

## The 10 additions

### #1 — Three Questions Framework

**Magnum Opus + consulting playbook:** new section "Three Questions That Must Be Answerable Without Running Code" — state ownership, feedback location, deletion blast radius. Mandatory at engagement kickoff (consulting playbook §1.5) and at every tier-ship gate (consulting playbook §6).

**/cook Phase 0.6 (engagement discovery):** add Q12.5 / Q12.7 / Q12.9:
- "Where will state live in this system? (database / files / external service / in-memory / multiple — name each)"
- "How will we know the system is working? (logs / metrics / alerts / dashboards / user feedback / silent until broken)"
- "What's the blast radius of the riskiest component? (everything fails / one feature degraded / data corruption / silent wrong-answer)"

**design.md required sections (SignalWorks engagements):** add three new mandatory sections — "State Architecture", "Feedback Architecture", "Deletion Blast Radius."

**code-correctness audit prompt:** add hunt category — "for each non-trivial module, can the operator answer the three questions WITHOUT running the code? If no, that's a finding."

### #2 — Anti-pattern A17: System Incoherence

Add to consulting playbook anti-patterns library:

```
A17. System incoherence — multiple unrelated concerns inhabit a single boundary.

Symptom: a file, module, or service contains multiple flows, multiple state owners,
multiple feedback channels — there is no single sentence that summarizes its purpose.
The 7000-line single-file failure mode from the video is one extreme; subtler versions
exist at all scales (a "utils" module that does 12 unrelated things; a service that
owns both auth and rate-limiting and metrics).

The signal is NOT line count. The signal is: can a reader summarize this boundary's
responsibility in one sentence? If you find yourself saying "and also..." multiple
times, the boundary is wrong.

Fix: decompose along single-responsibility lines. Extract distinct concerns to
separate boundaries. Update tests to exercise each in isolation. Update the
adjacent docs to reflect the new boundaries (anti-pattern A9 — adjacent layers
must be updated in lockstep).

General rule: every module, file, service has a single sentence that describes its
responsibility. If the sentence requires "and" or "also", the boundary is wrong.
```

### #3 — Jagged Frontier Section

Add to consulting playbook §3 (Authority Block Setup) — new subsection "Jagged Frontier — what NOT to delegate even within autonomous authority":

- Novel-domain architectural decisions (no priors to draw from)
- Security-critical algorithm choices (probabilistic output is unsafe)
- State machine design (correctness requires holistic invariant reasoning)
- Irreversible operations on production systems (financial transactions, deletions, schema migrations)
- Cross-system contracts where the LLM hasn't seen the full picture (API design, data format definitions)
- Anything where probabilistic behavior could compound into wrong-answer-at-scale

Add to magnum-opus.md as a universal principle (not just consulting).

### #4 — Design Before Prompt Mandate

Add to client-engagement-CLAUDE-template.md Required Rules + magnum-opus.md template:

> "Before invoking Claude on any substantive task (a tier ship, a new component, a refactor), sketch the components + data flows + state ownership + failure points. The sketch lives in `docs/plans/sketches/{date}-{topic}.md` (markdown OK, ASCII art OK, photo of whiteboard OK — the format is flexible, the discipline is not). The sketch is the operator's theory of the system before AI gets involved. Reference the sketch when prompting; let it constrain Claude's options. Update the sketch as the system evolves — stale sketches are worse than none."

### #5 — Theory-Building Reframing

Rewrite consulting playbook §4 (Source-of-Truth Doc Spine) intro:

> "These six docs are not bureaucracy. They are the *theory of the system* — the program in the operator's head, made explicit and shareable. Per Peter Naur's 1985 paper 'Programming as Theory Building,' the program is the mental model of how pieces connect; the code is merely its shadow. AI-era engagements require the theory to be explicit and shared, not implicit and trapped in one person's memory. When the theory lives in the doc spine, any operator (or fresh Claude session) can pick up cold. When it lives only in someone's head, the engagement dies if that person leaves."

Same intro added to magnum-opus.md Phase 4 (Scaffold Output).

### #6 — AI Decision Points + Verification Mechanisms

Add to design.md required sections (both magnum-opus + consulting playbook templates):

> "## AI Decision Points + Verification Mechanisms
>
> Every place AI output drives runtime behavior in this system, listed with the verification mechanism gating each. AI is a probabilistic translator (per the video summary); identifying these points explicitly is itself an architectural fact worth documenting."
>
> Format per entry:
> - **Decision point:** what AI is deciding (e.g., 'summarize source documents', 'classify support ticket', 'generate code suggestion')
> - **Verification mechanism:** how we know the output is acceptable (eval, grounding check, human approval, deterministic post-process, automated regression test, or 'none — accepted as best-effort')
> - **Failure mode:** what happens if AI is wrong (silent wrong-answer / degraded UX / hard error / data corruption)
> - **Reversibility:** can we undo the AI's decision after the fact?

### #7 — System Coherence Audit (revised from max-file-length)

Add to code-correctness audit prompt as hunt category:

> "**System coherence.** For each non-trivial module/file/service: can a reader summarize its responsibility in ONE sentence? If the summary requires 'and' or 'also', flag as P2 with anti-pattern A17 reference. The signal is NOT line count — big files with single coherent purpose are fine; small files straddling concerns are bad. Specifically check: are state owners co-located with the components that need them? Are feedback channels distinct from happy-path code? Is each public function's purpose self-evident from name + signature?"

### #8 — WHY Alongside WHAT in Spec Framework

Modify /cook Phase 1.5 (Specification + Pre-flight):

> "For each of the 7 spec properties, capture both:
> - **WHAT:** the requirement / behavior / contract
> - **WHY:** the user need it serves, the prior decision it enacts, or the constraint it satisfies
>
> No WHY → spec is incomplete; ask again. WHY-less specs decay into requirements lists that don't survive scope debates. The WHY is what allows future operators (and Claude) to make trade-off decisions when reality conflicts with the literal WHAT."

### #9 — Client Involvement Model Question

Add to /cook Phase 0.6 as new question Q11.5:

> "11.5. **Client involvement model?**
> - (a) Hands-off owner — Trent decides everything technical, client sees outcomes only
> - (b) Observer — kept informed at milestones, no technical input
> - (c) Collaborator — wants to weigh in on architectural choices
> - (d) Absent — minimal communication, async-only updates
>
> Affects: authority block (more conservative for c, more autonomous for a/d), communication cadence, how much 'explain the system' energy goes into client-facing artifacts, whether `docs/{client}-onboarding.md` is technical or executive."

### #10 — Patient-Senior-Dev Audit Posture

Add to each audit prompt (code-correctness, security, ux-delivery, docs-operations) preamble:

> "**Posture:** You are a patient senior developer reviewing this work, not a grader. Your purpose is to surface the system's underlying *theory* — why was it done this way? what if X were different? what assumption is load-bearing here? — not just check correctness against a checklist. Ask 'why' and 'what if' as you sweep. Findings that name the underlying theory gap (not just the symptom) are higher-value."

## Critical-review pass

1. **Risk: 10 additions is a lot for one patch.** Most are content additions to existing docs (cheap). Two require new design.md sections (#1, #6) which materially expand what /cook produces. One requires new convention (#4 sketches folder). Mitigation: bundle all under "v1.6 systems-thinking patch" with one cohesive narrative; trace each to the video summary line.
2. **Risk: Theory-building reframing might feel preachy.** The Naur quote is genuinely load-bearing, but operators may skim it. Mitigation: keep the reframing tight (~3-4 sentences), let it earn its place by being referenced from multiple sections.
3. **Risk: Sketches folder convention creates friction if operator skips it.** Mitigation: don't require sketches for trivial work (single-line bug fix, config change); only for "tier ship, new component, refactor" per the mandate.
4. **Risk: Three Questions framework duplicates the design.md structure.** Mitigation: explicitly position the three questions as the EARLY-STAGE diagnostic (Phase 0.6 + design.md sections); the deeper architecture decisions go into Phase 2 Harness Design as before.
5. **Risk: Magnum Opus + consulting playbook duplication.** Some content lives in both. Mitigation: Magnum Opus carries the universal framing; consulting playbook references back ("see magnum-opus §X for the universal version") for shared content. Specialized application stays in consulting playbook.
6. **Risk: Existing engagements (brett-gove-intell) have implicit answers to Three Questions but never made them explicit.** Out-of-scope to retro-fit; v1.6 only applies to NEW engagements scaffolded after this ships.
