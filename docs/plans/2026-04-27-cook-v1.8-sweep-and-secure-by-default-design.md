## Problem statement

Two related gaps in the current /cook + audit triad system:

**Gap 1 (Deployment gate is manual orchestration).** Multi-agent post-launch sweeps work — proven on brett delivery (consulting playbook §13). But orchestration is manual: operator copies audit prompts, fills `{{PROJECT_CONTEXT}}` / `{{SLICE}}` / `{{KNOWN_PRIORS}}`, dispatches 4 agents, consolidates findings. Operator's reported failure mode: "I dispatch agents, they find bugs, I fix, dispatch again, they find MORE bugs, continuous." Each manual cycle has limited slice context; new cycles surface bugs prior cycles couldn't see. Under deadline pressure, the orchestration overhead gets cut and sweeps get skipped.

**Gap 2 (Security is a phase, not a posture).** /cook v1.5+ has Phase 0.9 (Security Pre-flight, 6 yes/no checks) + Phase 5 (first-pass threat model) + security-reviewer agent deployment. But security work happens "later" — scaffolds ship with weak defaults that the operator is supposed to harden manually. For a system whose standard is "public-deploy-ready, attack-safe across the board" (consulting AND personal), this leaves too many opportunities to forget.

## Goals

1. **Build `/sweep` skill** — one-command multi-agent deployment gate. 9-11 agents in 2 waves + verification round. Auto-detects project type. Loads heavy KB context per agent. Hard pass-gate (GO/HOLD/NO-GO). Catches all P1 issues in one sweep.
2. **Make /cook secure-by-default** — security woven into Phase 0.6 (threat-model questions), Phase 1 (exposure-level routing), Phase 1.5 (WHO-COULD-EXPLOIT-THIS for every WHAT/WHY), Phase 2 (AI-project security architecture decisions), Phase 4 (strong default scaffolded files), Phase 5 (threat-model.md artifact mandate).
3. **CLAUDE.md mandate addition** — `/sweep` invocation before any deploy declaration, parallel to existing session-handoff mandate.
4. **Update consulting-playbook.md** to reference new sweep + secure-by-default posture.
5. **Update MEMORY.md + KB-INDEX.md** for v1.8 system.

## Non-goals (deliberately out of scope)

- Automated FIXING of /sweep findings (operator approves each fix).
- Continuous runtime monitoring (this is deploy-time, not runtime).
- Third-party penetration testing replacement.
- Backporting to existing engagements (brett-gove-intell stays as-is; v1.8 applies forward).
- Replacing `pre-ship` skill (extend, don't deprecate; keep human-readable checklist for quick solo runs).

## Architecture

### `/sweep` skill — 9-11 agents, 2 waves + verification

**Auto-detect project type** by inspecting:
- `package.json` / `pyproject.toml` / `Cargo.toml` → language stack
- `index.html` / `src/` / `public/` → web app
- `signalworks_takeaways_target:` in CLAUDE.md → SignalWorks engagement
- `docs/plans/threat-model.md` → exposure level
- `docs/audit-protocol.md` → audit triad scaffolded
- Live URL → from CLAUDE.md or operator override

**Wave 1 (parallel, 7 agents):**

Each agent gets a brief that loads relevant KB context inline (NOT just generic prompt). Briefs include: project context, the hard mandate ("DEPLOYMENT GATE — findings you miss become production bugs"), stakes context, pre-loaded checklists from KB, adversarial framing ("try to BREAK this"), self-critique mandate.

| # | Agent | Territory | KB context loaded |
|---|---|---|---|
| 1 | security-attack-surface | OWASP LLM Top 10, 6 attack classes, indirect prompt injection vectors | `ai-security.md` §3-§4, anti-patterns A13/A18 |
| 2 | security-deployment | pre-ship Security checklist + secrets/credentials/auth/CORS/HTTPS | `pre-ship` SKILL.md Security section, anti-pattern A3 |
| 3 | code-correctness | A6/A9/A18/A19 + System Coherence (A17) + Three Questions answerable | code-correctness audit prompt + A1-A20 catalog |
| 4 | infrastructure-deployment | pre-ship Infra/Database/Monitoring + 4 SaaS failure modes | pre-ship + `building-ai-saas.md` |
| 5 | data-integrity | content_hash contract + silent failures (A4) + zombie tests (A6) | consulting playbook §7 + ingestion contract |
| 6 | ux-client-ready | A10 (bare-dashboard) + A16 (ship-ready vs client-ready) + first-impression-trust | ux-delivery audit prompt + anti-patterns |
| 7 | docs-operability | runbook completeness + decision-log freshness + operator commands | docs-operations audit prompt |

**Conditional Wave 1 agents (classification-gated):**

| # | Agent | Fires when | KB context |
|---|---|---|---|
| 8 | ai-eval-integrity | Project type = Type A (AI deliverable) | `evaluation.md` + hallucination prevention §16.2 |
| 9 | ai-rag-security | Project uses RAG | `ai-security.md` §4 RAG-specific attack surfaces |

**Wave 2 (sequential after Wave 1, 1 verifier agent):**

10. **cross-pollination-verifier** — receives ALL Wave 1 findings as input. Hunts for bugs that REQUIRE cross-territory reasoning. Examples: "security found prompt-injection vector X; does code-correctness's finding Y compound that risk?" Catches what slips through territory-isolation.

**Verification round (sequential after Wave 2):**

11. **deletion-test-agent** — runs Three Questions framework against EVERY P1 finding: "if we ship this fix, what NEW component does it depend on? Where does its state live? What breaks if we revert it?" Catches fix-introduces-new-bug cycles.

**Pass gate / output:**

- **HARD FAIL (blocks deploy):** Any P1 from security-attack-surface, security-deployment, data-integrity, or cross-pollination-verifier
- **SOFT FAIL (warns):** P2 findings — operator decides
- **PASS:** All Wave 1+2 P1s clean OR explicitly waived by operator with documented justification

Output: `.sessions/sweeps/sweep-{date}-{commit-sha}.md` with verdict (GO / HOLD / NO-GO) at top, P1/P2/P3 findings grouped by territory, cohort split (re-run-needed vs no-re-run), auto-update next handoff's "Audit Directions" slot.

**Wall-clock estimate:** ~25-30 min (Wave 1 parallel ~15-20 min; Wave 2 + verification sequential ~10 min).

### /cook security-by-inception patches

**Phase 0.6 — 4 new threat-model questions (after Q11 Operational Constraints, before Q11.5 Client Involvement):**
- Q11.6: Exposure level (public / auth-gated / internal / local-only)
- Q11.7: Sensitive data handled? (PII / money / credentials / health / legal — or none)
- Q11.8: Worst-case if compromised? (data breach / financial loss / reputational damage / legal liability / minor inconvenience)
- Q11.9: Realistic threat actors? (script kiddies / competitors / nation-state / nobody-realistic)

**Phase 1 — exposure-level routing (in classification report):**
- Public → mandates `/sweep` before any deploy + scaffolds attack-surface defaults + auth/rate-limit middleware
- Auth-gated → scaffolds auth middleware + session security + rate-limiting on auth endpoints
- Internal → basic hygiene (secrets, gitignore, dep audit)
- Local-only → minimum hygiene only

**Phase 1.5 — WHO-COULD-EXPLOIT-THIS:**
- For each of the 7 spec properties, the WHAT + WHY + new third question: "if an attacker controlled this input/output/state, what could they do?"
- Spec is incomplete if any of the three is missing.

**Phase 2 — mandatory security architecture decisions (Type A AI projects):**
- Prompt injection defense strategy (spotlighting per `ai-security.md` §4? input validation? taint tracking?)
- Indirect injection vectors (does the agent read external content?)
- Output sanitization (URL scheme allowlist, HTML escaping, JSON validation)
- Tool/permission scoping (least privilege per agent action)
- Reference: `ai-security.md` §3-§4 + OWASP LLM Top 10

**Phase 4 — strong defaults baked into scaffold:**

ALL projects:
- Pre-commit hook config: secret scanning (gitleaks)
- `.github/dependabot.yml` (or equivalent) for dependency vulnerabilities
- `.gitignore` covering `.env`, credentials, common cred file patterns
- `.env.example` with placeholders only

Web/public projects (gated by Phase 1 = public or auth-gated):
- Security headers config (CSP, HSTS, X-Frame-Options, X-Content-Type-Options)
- Auth middleware skeleton (if auth-touching)
- Rate limiting middleware skeleton (for public endpoints)
- Input validation patterns scaffold (pydantic / zod / equivalent)
- HTTPS-only enforcement scaffolded
- CORS scoped to specific origins (no wildcards)

AI projects (Type A):
- Spotlighting prompt template (Microsoft's >50% → <2% injection defense)
- Output validation gates scaffold (taint tracking helpers)
- Tool permission registry scaffold (least-privilege scoping)
- Red-team test scaffold (`tests/red_team/` with sample injection inputs)

Data-handling projects:
- Logging config excludes PII (structured JSON logs with PII redaction, leveraging existing `secret_redactor.py` stub)
- DB connection scaffolding uses parameterized queries
- Backup + restore-test mandate in CLAUDE.md

**Phase 5 — threat-model.md artifact mandate:**

Every scaffold produces `docs/plans/threat-model.md` with structured sections:
- Actors (who could attack)
- Assets (what they want — data, money, reputation, compute)
- Threats (specific attack vectors per Phase 0.6 answers)
- Defenses (what's scaffolded, what's deferred)
- Residual risks (known gaps with rationale)

NOT a 50-page doc — a 1-page structured artifact answering Phase 0.6 + 0.9 questions formally.

**CLAUDE.md mandate addition (in client-engagement-CLAUDE-template.md):**
- "Before declaring deploy-ready, invoke `/sweep`. Not optional."
- Parallel to session-handoff mandate.

## Critical-review pass

1. **Risk: 9-11 agents per sweep is heavy on tokens/wall-clock.** Real cost ~30 min wall + parallel sub-agent context. Mitigation: that's still cheaper than multi-cycle alternative; bugs caught are higher severity; operator can run quick `pre-ship` for non-deploy-day checks.
2. **Risk: zero-bug-in-one-sweep is impossible.** Mitigation: explicit framing — aim for "all P1 caught, only P2/P3 surface in subsequent sweeps." Document the bar honestly.
3. **Risk: false-positive P1s trigger HOLD unnecessarily.** Mitigation: trust-but-verify discipline (consulting playbook §7) — every P1 finding gets one-liner verification in cross-pollination round. Operator can manually waive with documented justification.
4. **Risk: AI-specific agents are deadweight on non-AI projects.** Mitigation: classification-gated dispatch (only fire if project type matches).
5. **Risk: `pre-ship` and `/sweep` overlap.** Mitigation: clear roles — `pre-ship` = quick human checklist; `/sweep` = heavy multi-agent for going-public gates. `/sweep` explicitly references `pre-ship` SKILL.md content in its security-deployment agent brief.
6. **Risk: /cook security-by-default scaffolds files operators don't always need (e.g., auth middleware on a Type B static site).** Mitigation: classification-gated scaffolding — only generates files appropriate to project type.
7. **Risk: Phase 0.6 question count growing (now 11 + 4 new = 15 questions).** Mitigation: each new question is yes/no/multiple-choice; estimated added time ~2 min. Worth the upfront cost.
8. **Deliberately NOT in scope:** auto-fix, runtime monitoring, replacing pen tests, backporting.
